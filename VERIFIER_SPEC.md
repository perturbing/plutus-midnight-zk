```

  Ōö£ŌöĆ Transcript ŌåÆ x, xŌéü, xŌéé, xŌéā, xŌéä, y, ╬Ė, ╬▓, ╬│, trashChal
```

## 2. Notation and field arithmetic

| Symbol | Meaning |
|--------|---------|
| `F_q` | BLS12-381 scalar field, prime `q = 52435875175126190479447740508185965837690552500527637822603658699938581184513` |
| `F_p` | BLS12-381 base field (used for G1/G2 coordinates) |
| `G1`, `G2` | BLS12-381 groups; additive notation |
| `G1_0` | G1 identity (point at infinity); compressed form = 48 zero bytes with high bit set |
| `[a]Ōéü` | `a ┬Ę G1_gen` |
| `[s]Ōéé` | `s ┬Ę G2_gen` (from SRS) |
| `e(P, Q)` | Optimal Ate pairing; `bls12_381_millerLoop` + `bls12_381_finalVerify` in Plutus |
| `LE(bs)` | Interpret byte string `bs` as a little-endian non-negative integer |
| `msm(ss, Ps)` | Multi-scalar multiplication `╬ŻßĄó sßĄó ┬Ę PßĄó`; `bls12_381_G1_multiScalarMul` |
| `blake2b_256(bs)` | 32-byte BLAKE2b-256 hash of `bs` |
| `Scalar n` | Field element `n mod q` |
| `╬┤` | BLS12-381 scalar field coset generator: `╬┤ = 7^{2^32} mod q = 3793952369011177517951424454785176000433849974408744014172535497121832470999`. Hardcoded as `bls12_381_scalar_delta` in `BlsUtils.hs`. See ┬¦6.3. |

**Field structure note (why ╬┤ works as a coset generator):**
`F_q*` is cyclic of order `q ŌłÆ 1 = T ┬Ę 2^32` with `T` odd, generated by `7`.
Since `gcd(T, 2^32) = 1`, the Chinese Remainder Theorem gives an internal direct product:
```
F_q* Ōēģ Ō¤©ŽēŌ¤® ├Ś Ō¤©╬┤Ō¤®
```
where `Žē = 7^T` has order `2^32` and `╬┤ = 7^{2^32}` has order `T`.
The two subgroups share only the identity `{1}`.

Let `H = {1, Žē^{2^32/n}, Žē^{2┬Ę2^32/n}, ŌĆ”}` be the evaluation domain of size `n = 2^k`.
The cosets `╬┤^0┬ĘH, ╬┤^1┬ĘH, ╬┤^2┬ĘH, ŌĆ”` are pairwise disjoint: if `╬┤^i┬ĘhŌéü = ╬┤^j┬ĘhŌéé` for
`hŌéü,hŌéé Ōłł H` then `╬┤^{iŌłÆj} = hŌéé┬ĘhŌéüŌü╗┬╣ Ōłł H`, but `╬┤^{iŌłÆj}` has odd order and every
element of `H` has 2-power order, so both sides must be `1`, giving `i = j`.
The permutation argument encodes cell `(column i, row j)` with the unique identity
`╬┤^i ┬Ę Žē^j`, so a prover cannot conflate values across different columns.

All hex strings in the JSON are **little-endian**: byte 0 is the least significant byte.

Field inversion uses the new modular exponentiation plutus builtin via
`expModInteger a (-1) q` in Plutus.

## 3. JSON test vector formats

For human readability all artifacts are written to JSON, as it is more transparent than raw bytes.

### 3.1 Plutus VK JSON

File: `*_plutus_vk.json`. Produced by `write_json_all_artifacts` in `circuit_params.rs`.

Contains the **trusted-setup-dependent** fields only: the polynomial commitments, SRS G2
point, and the domain parameters derived from the circuit's final compiled form.  These
fields change when you redo the trusted setup (new SRS), but are otherwise fixed for a
given circuit.

All G1 commitments are 48-byte compressed BLS12-381 points encoded as hex strings.
All scalars are 32-byte little-endian field elements encoded as hex strings.

```json
{
  "k":                     <integer>,        // logŌéé(domain size); N = 2^k
  "omega":                 "<32-byte LE hex>", // primitive N-th root of unity Žē
  "blinding_factors":      <integer>,        // number of blinding rows (5 or 6 in the test circuits)
  "num_advice_columns":    <integer>,        // na
  "num_perm_columns":      <integer>,        // npc (= len(permutation_commitments))
  "cs_degree":             <integer>,        // constraint system degree d; chunk_size = d-2, nh = d-1
  "num_lookups":           <integer>,        // nl
  "fixed_commitments":     ["<48-byte hex>", ...],            // one per fixed column
  "permutation_commitments": ["<48-byte hex>", ...],          // npc entries (sigma polys)
  "s_g2":                  "<96-byte hex>",  // [s]GŌéé from SRS
  "transcript_repr":       "<32-byte hex>"   // VK identity hash (Fiat-Shamir seed)
}
```

**What changes with a new SRS**: `fixed_commitments`, `permutation_commitments`, `s_g2`,
and `transcript_repr` are all derived from the SRS randomness.  `k`, `omega`,
`blinding_factors`, and the scalar dimensions are fixed by the circuit design.

**Why `advice_queries` / `fixed_queries` are absent**: These arrays are committed to via
`transcript_repr` on the Rust side, so they are implicitly bound to the Fiat-Shamir
challenges.  The verifier never needs to read them directly: evaluation indices are
pre-computed into `eval_idxs` in `*_rotation_sets.json`, and `naq` / `nfq` are available
in `*_circuit_params.json`.

### 3.2 Circuit constraint JSON

File: `*_circuit_constraint.json`. Produced by `write_json_all_artifacts` in `circuit_params.rs`.

Contains the **circuit-design-dependent** fields: the symbolic polynomial expressions
that encode the constraint system.  These change only when the circuit logic changes,
not when the trusted setup (SRS) is redone.  Analogous to `*_rotation_sets.json`, which
is also circuit-design-dependent and in its own file.

```json
{
  "gate_polys":             [ <instr_array>, ... ],  // one RPN instruction array per gate poly
  "gate_sel_cols":          [ <uint> | null, ... ],  // parallel to gate_polys (see below)
  "perm_col_types":         [{"col_type":<0|1|2>,"eval_idx":<int>}, ...],  // npc entries
  "lookup_input_exprs":     [ [ [<instr_array>, ...], ... ], ... ],  // nl ├Ś chunk ├Ś parallel_input ├Ś width
  "lookup_table_exprs":     [ [<instr_array>, ...], ... ],  // nl ├Ś width_exprs (for ╬Ė-compression)
  "lookup_selector_exprs":  [ <instr_array>, ... ],  // nl entries ŌĆö one selector per lookup
  "trash_selectors":        [ <instr_array>, ... ],  // one per trashcan argument
  "trash_constraint_exprs": [ [<instr_array>, ...], ... ]  // one list of exprs per trashcan
}
```

**`gate_sel_cols`** is a new field parallel to `gate_polys`. For each gate polynomial, the
value is either the fixed **column index** (into `vkFixedComs`) of the simple selector that
gates this polynomial, or `null` if the gate uses no simple selector. Simple selectors are
boolean polynomials that are 0 or 1 on every row; midnight-proofs does not include them in
the multi-open proof stream and instead handles their commitment via the linearization
polynomial `lin_com`. The verifier uses `gate_sel_cols` to compute the linearization
commitment evaluation `linComEval` (see ┬¦7.2 and ┬¦6.2).

**`lookup_input_exprs`** has a 4-level nested structure for the LogUp argument:
`[lookup][chunk][parallel_input][width_exprs]`. Each innermost array encodes one expression
in RPN format (see ┬¦3.3). A "chunk" groups input expressions whose combined degree stays
within `cs_degree`; "parallel inputs" are the multiple inputs that are ╬Ė-compressed together.

**`lookup_selector_exprs`** contains one RPN expression per lookup that evaluates to 1 on
rows where the lookup is active and 0 elsewhere.

**`perm_col_types`** encodes how to look up each permutation column's evaluation at `x`:

| `col_type` | Column type | Evaluation source |
|------------|-------------|-------------------|
| `0` | Advice | `prfAdviceEvals[eval_idx]` |
| `1` | Fixed | `prfFixedEvals[eval_idx]` |
| `2` | Instance | `instEvals[eval_idx]` |

`eval_idx` is the position in the corresponding evaluation list at `Rotation::cur()` (i.e.,
the query at rotation 0 for that column), as it appears in `cs.advice_queries()` /
`cs.fixed_queries()` / `cs.instance_queries()`.

**Note on ╬┤ (delta):** The coset generator ╬┤ used in the permutation argument (see ┬¦6.3) is
**not** stored in this file.  It is a pure constant of the BLS12-381 scalar field ŌĆö
`╬┤ = 7^{2^32} mod q` ŌĆö and is hardcoded as `bls12_381_scalar_delta` in `BlsUtils.hs`.

All instruction arrays (`gate_polys`, `lookup_*_exprs`, `trash_*`) use the RPN format
described in ┬¦3.3.

### 3.3 Gate polynomial RPN instruction format

All symbolic polynomial fields in `*_circuit_constraint.json` ŌĆö `gate_polys`,
`lookup_input_exprs`, `lookup_table_exprs`, `trash_selectors`, `trash_constraint_exprs` ŌĆö
are stored as **flat RPN instruction arrays** ŌĆö sequences of JSON objects, each with an
`"op"` field, encoding a polynomial expression in postfix order.

`instrsToGateExpr` in `JsonParser.hs` converts each array into a `GateExpr` recursive
expression tree (see ┬¦6.1) by simulating the RPN stack at parse time with `GateExpr` nodes
as stack values.  Constants and scale factors are decoded from hex at parse time and stored
directly in the tree ŌĆö no byte-scanning occurs at on-chain evaluation to save on the plutus budget.

#### Instruction objects

| JSON object | `GateExpr` constructor |
|-------------|------------------------|
| `{"op":"Constant","value":"<32-byte LE hex>"}` | `GEConst (decoded Scalar)` |
| `{"op":"Advice","query_index":<uint>}` | `GEAdv qi` |
| `{"op":"Fixed","query_index":<uint>}` | `GEFix qi` |
| `{"op":"Instance","query_index":<uint>}` | `GEInst qi` |
| `{"op":"Negated"}` | `GENeg a` |
| `{"op":"Sum"}` | `GEAdd a b` |
| `{"op":"Product"}` | `GEMul a b` |
| `{"op":"Scaled","factor":"<32-byte LE hex>"}` | `GEScale a (decoded Scalar)` |

`query_index` is an index into the corresponding evaluation list, in the order those queries
appear in the circuit's `advice_queries()` / `fixed_queries()` / `instance_queries()`.

In Halo2, `Selector` is a boolean-like column used to conditionally enable gates on specific rows.
During `keygen_vk`, the compiler multiplies each selector value into the gate polynomial expressions
as a fixed column, eliminating `Selector` nodes from the symbolic representation entirely. By the
time the verifying key is finalised, only `Advice`, `Fixed`, `Instance`, `Constant`, `Sum`,
`Product`, `Negated`, and `Scaled` nodes remain. A `Selector` instruction in the JSON would
indicate that the upstream Rust compilation pipeline failed to compile it away ŌĆö a bug in circuit
key generation, not something `instrsToGateExpr` is expected to handle.

midnight-zk circuits have no multi-phase challenges (though it can support it), so the `Challenge` instruction will
not appear in practice and is not handled by `instrsToGateExpr`. E.g, this is a single round
verifier, the prover sends only one message to the verifier (the smart contract).

#### Example

The expression `(Fixed[9] + Advice[0]) * Constant[c] ŌłÆ Advice[1]` encodes as:

```json
[
  {"op": "Fixed",    "query_index": 9},
  {"op": "Advice",   "query_index": 0},
  {"op": "Sum"},
  {"op": "Constant", "value": "..."},
  {"op": "Product"},
  {"op": "Advice",   "query_index": 1},
  {"op": "Negated"},
  {"op": "Sum"}
]
```

`instrsToGateExpr` builds the tree
`GEAdd (GENeg (GEAdv 1)) (GEMul (GEConst c) (GEAdd (GEFix 9) (GEAdv 0)))`
reading from the stack state after processing each instruction in postfix order.

> **Why an ADT instead of keeping RPN?**
> Haskell's algebraic data types enable structural recursion (a simple `go` catamorphism
> in `evalGate`) rather than an explicit stack machine. This eliminates the partial error
> cases that a runtime stack needs (underflow, leftover values), makes the evaluator a
> total function, and plays well with PlutusTx serialisation via `makeIsDataIndexed`.
> The RPN-to-tree conversion is paid once at parse time inside `instrsToGateExpr`; the
> on-chain evaluator never sees raw instruction arrays.

### 3.4 Circuit params JSON

File: `*_circuit_params.json`. Produced by `write_json_circuit_artifacts`.

> **Note:** this file is not read by the Haskell parser or verifier. All dimensions it lists
> are either already present in the other JSON artifacts or derived at runtime from array
> lengths. Its purpose is human-readable reference: a compact snapshot of circuit dimensions
> useful for auditing, comparing circuits side-by-side, or benchmarking (e.g. tracking how
> `num_advice_columns` or `num_rotation_sets` change across circuit versions).

```json
{
  "num_advice_columns":           <integer>,  // na ŌĆö advice (witness) columns
  "num_lookups":                  <integer>,  // nl ŌĆö lookup arguments
  "num_permutation_commitments":  <integer>,  // npc ŌĆö columns covered by permutation
  "permutation_chunk_size":       <integer>,  // chunk_size = cs_degree ŌłÆ 2
  "num_quotient_commitments":     <integer>,  // nh = cs_degree ŌłÆ 1
  "num_advice_queries":           <integer>,  // naq ŌĆö total advice evaluations in proof
  "num_fixed_queries":            <integer>,  // nfq ŌĆö total fixed evaluations in proof
  "num_blinding_commitments":     <integer>,  // ntrash ŌĆö extra G1 blinding points in the proof (0 or 1)
  "num_permutation_product_evals":<integer>,  // nppe = (npŌłÆ1)├Ś3 + 2
  "num_rotation_sets":            <integer>   // nq ŌĆö must equal num_sets in rotation_sets.json
}
```

**`blinding_factors`** (in the VK) is the number of blinding rows appended to each column
in the witness. It controls the last-usable-row offset (`x_last = x ┬Ę Žē^{-(blinding+1)}`)
and the Lagrange polynomials `lLast` and `lBlind` in the constraint check. The test circuits
use `blinding_factors` of 5 or 6 (see ┬¦10).

**`num_blinding_commitments`** (in circuit_params.json) is a separate quantity: the count of
extra G1 blinding commitment points (`prfTrashComs`) appended to the proof transcript. It is
determined from the actual proof size, not directly from `blinding_factors`. A circuit with
`blinding_factors = 6` may still have `num_blinding_commitments = 0` (e.g. Ethereum ECDSA).

### 3.5 Rotation sets JSON

File: `*_rotation_sets.json`. Produced by `rotation_sets_bytes()` in `rotation_sets.rs`.

```json
{
  "num_sets": <integer>,
  "sets": [
    {
      "rotations": [<i32>, ...],       // rotation offsets: 0, ┬▒1, -(blinding+1)
      "slots": [
        {
          "poly_kind": "<string>",     // see poly-kind table below
          "index":     <integer>,      // primary index (meaning depends on poly_kind)
          "eval_idxs": [<integer>, ...]  // one per rotation; used by Advice and Fixed only
        },
        ...
      ]
    },
    ...
  ]
}
```

Sets are in GWC sorted order: ascending cardinality, ties broken by first-encounter index.
Slots within a set are in xŌéü-power order (the order they are multiplied by powers of xŌéü).

#### Poly-kind table

| `poly_kind` | `index` meaning | Commitment | Evaluation |
|-------------|-----------------|-----------|------------|
| `"Advice"` | advice column index | `prfAdviceComs[index]` | `prfAdviceEvals[eval_idxs[rotPos]]` |
| `"Instance"` | (ignored ŌĆö see note) | G1 identity | `zero` (hardcoded) |
| `"LogupMult"` | lookup index k | `prfLookupMultComs[k]` | `prfLogupEvals[eval_idxs[rotPos]]` |
| `"Trash"` | trash index k | `prfTrashComs[k]` | `prfTrashEvals[eval_idxs[rotPos]]` |
| `"Fixed"` | fixed column index | `vkFixedComs[index]` | `prfFixedEvals[eval_idxs[rotPos]]` |
| `"PermSigma"` | sigma column index | `vkPermSigmaComs[index]` | `prfPermSigmaEvals[eval_idxs[rotPos]]` |
| `"H"` | (ignored) | lin_com commitment (see ┬¦7.2) | `linComEval` at RotCur; dummy evals otherwise |
| `"PermProd"` | chunk index j | `prfPermProdComs[j]` | `prfPermProdEvals[eval_idxs[rotPos]]` |
| `"LogupAccum"` | lookup index k | `prfLookupAccumComs[k]` | `prfLogupEvals[eval_idxs[rotPos]]` |
| `"LogupHelper"` | flat helper index | `prfLookupHelperComs[index]` (flat concat) | `prfLogupEvals[eval_idxs[rotPos]]` |

> **Why `index` is ignored for `Instance` and `H`:**
>
> - **`Instance`**: midnight-zk never commits to an instance column in the normal sense.
>   Instance column 0 is the zero polynomial (public inputs are handled via Lagrange
>   interpolation on a separate column), so its commitment is always the G1 identity and its
>   evaluation is always 0. Both values are fixed by convention; nothing is read from the proof
>   and `index` is meaningless.
>
> - **`H`** (linearization commitment): `lin_com` is built from `nh` h-piece commitments
>   `prfHComs[0..nhŌłÆ1]` **plus** the simple-selector column commitments from `vkFixedComs`.
>   There is no single indexed commitment to pick. Its evaluation `linComEval` is derived by
>   the verifier from the gate constraint sum (see ┬¦7.2). Both the commitment MSM and the
>   evaluation bypass the `index` field entirely.

For `"Advice"`, `"Fixed"`, `"LogupMult"`, `"LogupAccum"`, `"LogupHelper"`, `"PermProd"`,
`"PermSigma"`, and `"Trash"`: `eval_idxs[rotPos]` is an absolute offset into the unified
evaluation array:

```
unifiedEvals = adviceEvals | fixedEvals | sigmaEvals | permProdEvals |
               logupEvals  | trashEvals | dummyEvals
```

For `"H"` at `RotCur`, the eval is `linComEval` (not from the proof). At other rotations,
`eval_idxs[rotPos]` indexes into the `dummyEvals` tail of `unifiedEvals` ŌĆö these are the
fewer-point-sets dummy evaluations generated by the prover for the merged rotation set.

For `"Instance"`, `eval_idxs` is unused; commitment and eval are always zero.

#### 3.5.1 PermProd eval layout

`prfPermProdEvals` is flattened to a list of length `(npŌłÆ1) ├Ś 3 + 2`:

- Non-last chunk `j` (0..npŌłÆ2): 3 evals = Z_j(x), Z_j(xŽē), Z_j(xŽē^last) ŌåÆ indices `3j`, `3j+1`, `3j+2`
- Last chunk (index npŌłÆ1): 2 evals = Z_{npŌłÆ1}(x), Z_{npŌłÆ1}(xŽē) ŌåÆ indices `3(npŌłÆ1)`, `3(npŌłÆ1)+1`

The `eval_idxs` in `rotation_sets.json` for `"PermProd"` slots contain the **absolute offsets
into `unifiedEvals`** (not chunk-relative indices). Each rotation position in the set maps to
the precomputed absolute offset for that (chunk, rotation) pair.

### 3.6 Proof JSON

File: `*_plutus_proof.json`. Produced by `proof_to_json` in `circuit_params.rs`.

All G1 points are 48-byte hex strings. All scalar evaluations are 32-byte LE hex strings.

```json
{
  "advice_commitments":              ["<48-byte hex>", ...],  // na entries [a_i]Ōéü
  "lookup_multiplicity_commitments": ["<48-byte hex>", ...],  // nl entries [m_k]Ōéü (one per lookup)
  "permutation_product_commitments": ["<48-byte hex>", ...],  // np entries [Z_j]Ōéü
  "lookup_logup_commitments": [                               // nl entries, one per lookup k
    {
      "helpers":     ["<48-byte hex>", ...],  // nc_k entries: [h_{k,0}]Ōéü ŌĆ” [h_{k,nc_kŌłÆ1}]Ōéü
      "accumulator": "<48-byte hex>"          // [Z_k]Ōéü
    },
    ...
  ],
  "trash_commitments":               ["<48-byte hex>", ...],  // ntrash entries [r_t]Ōéü
  "h_commitments":                   ["<48-byte hex>", ...],  // nh entries [h_j]Ōéü

  "advice_evals":     ["<32-byte hex>", ...],  // naq entries, in cs.advice_queries() order
  "fixed_evals":      ["<32-byte hex>", ...],  // nfq_total entries; simple selectors set to LE(1)
  "sigma_evals":      ["<32-byte hex>", ...],  // npc entries Žā_i(x)
  "permutation_product_evals": [               // np objects
    {"eval": "<hex>", "next_eval": "<hex>", "last_eval": "<hex>" | null},
    ...
  ],
  "lookup_evals": [                            // nl objects
    {
      "mult_eval":       "<32-byte hex>",      // m_k(x)
      "helper_evals":    ["<32-byte hex>", ...],  // nc_k entries: h_{k,0}(x) ŌĆ” h_{k,nc_kŌłÆ1}(x)
      "accum_eval":      "<32-byte hex>",      // Z_k(x)
      "accum_next_eval": "<32-byte hex>"       // Z_k(x┬ĘŽē)
    },
    ...
  ],
  "trash_evals":      ["<32-byte hex>", ...],  // ntrash entries r_t(x)
  "dummy_evals":      ["<32-byte hex>", ...],  // fewer-point-sets dummy evals (may be empty)
  "gwc": {
    "f_commitment": "<48-byte hex>",           // [f]Ōéü ŌĆö GWC auxiliary polynomial
    "q_evals":      ["<32-byte hex>", ...],    // nq entries: prover's q_i(xŌéā)
    "w_commitment": "<48-byte hex>"            // [w]Ōéü ŌĆö KZG opening witness ŽĆ
  }
}
```

**`fixed_evals`**: the serialiser reads `nfq_actual = nfq_total ŌłÆ num_simple_selectors` values
from the raw proof, then inserts the LE encoding of 1 at every simple-selector position
(determined by `simple_selector_mask` in `*_plutus_vk.json`) to reconstruct the full
`nfq_total`-length array. The Haskell verifier filters simple-selector positions back out when
absorbing into the transcript (they are not sent by the prover), but uses the full array
(with substituted 1s) when evaluating gate expressions via `fixEvals[qi]`.

**`permutation_product_evals`**: for the last chunk `last_eval` is `null`. All other chunks
include `last_eval` = evaluation at `x┬ĘŽē^last`. `parsePlutusProof` flattens this into
`prfPermProdEvals` (see ┬¦3.5.1).

**`lookup_evals` flatten order** in `prfLogupEvals`: for lookup k with nc_k chunks,
offset(k) = ╬Ż_{i<k}(nc_i + 3):
```
[offset+0]:          mult_eval  m_k(x)
[offset+1..offset+nc]: helper_evals  h_{k,0}(x) ŌĆ” h_{k,nc_kŌłÆ1}(x)
[offset+nc+1]:       accum_eval  Z_k(x)
[offset+nc+2]:       accum_next_eval  Z_k(x┬ĘŽē)
```

**`dummy_evals`**: present when the circuit was compiled with the `fewer-point-sets` feature.
These are synthetic evaluations appended at the end of the scalar stream (after `trash_evals`,
before the GWC `f_commitment`). They are absorbed into the Fiat-Shamir transcript as part of
`allEvals` before xŌéü/xŌéé are squeezed, and are referenced by `eval_idxs` entries in the
rotation set for the H slot at non-current rotations.

**`instance_poly_eval`**: omitted from the JSON. The committed instance polynomial is the
zero polynomial in midnight-zk (col 0 is always G1 identity), so its evaluation is 0. The
32 bytes are consumed from the raw proof stream to maintain offset alignment but are not
written to JSON. The verifier absorbs the constant 0 at this transcript position.

## 4. Transcript (Fiat-Shamir)

The transcript is a `BuiltinByteString` state, initialised to `vkTranscriptRepr` (32 bytes).
`absorb state bytes = state <> bytes` (append, no hashing).
`squeeze state = (Scalar (LE(blake2b_256 state) mod q), blake2b_256 state)` ŌĆö the state is
replaced by the 32-byte hash, not appended.

This matches the midnight-zk Rust prover: `transcript_data = blake2b_256(transcript_data)`.

Absorption order (must exactly match the midnight-zk v7 LogUp prover):

```
stateŌéĆ = vkTranscriptRepr                            // 32 bytes ŌĆö circuit identity
stateŌéü = stateŌéĆ <> G1_compressed_zero               // 48 bytes ŌĆö instance placeholder
stateŌéé = absorb stateŌéü (LE_32(len(pubInputs)))
stateŌéā = foldl (\s n -> s <> LE_32(n)) stateŌéé pubInputs

-- Advice commitments ŌåÆ ╬Ė (lookup ╬Ė-compression challenge)
stateŌéä = foldl (<>) stateŌéā prfAdviceComs            // na ├Ś 48 bytes
(╬Ė, stateŌéäs) = squeeze stateŌéä

-- Multiplicity commitments ŌåÆ ╬▓, ╬│ (permutation + LogUp challenges)
stateŌéģ = foldl (<>) stateŌéäs prfLookupMultComs       // nl ├Ś 48 bytes
(╬▓, stateŌéģb) = squeeze stateŌéģ
(╬│, stateŌéģg) = squeeze stateŌéģb
-- Note: ╬▓ is used in both the permutation argument and the LogUp constraint.
--       ╬│ is used only in the permutation argument.

-- Permutation z-product commitments (no squeeze here)
stateŌéå = foldl (<>) stateŌéģg prfPermProdComs         // np ├Ś 48 bytes

-- Per-lookup LogUp commitments: for each lookup k absorb helpers then accumulator
stateŌéć = foldl (\s (helpers, accum) -> foldl (<>) s helpers <> accum)
               stateŌéå
               (zip prfLookupHelperComs prfLookupAccumComs)
(trashChal, stateŌéćs) = squeeze stateŌéć               // squeezed AFTER all perm+logup coms

-- Trash commitments ŌåÆ y (gate Horner-folding challenge)
stateŌéćt = foldl (<>) stateŌéćs prfTrashComs
(y, stateŌéłs) = squeeze stateŌéćt

-- h-piece commitments ŌåÆ x (shared evaluation point)
stateŌéē = foldl (<>) stateŌéłs prfHComs
(x, stateŌéēs) = squeeze stateŌéē

-- Committed instance eval: always 0 in midnight-zk (absorb constant 0)
stateŌéēi = absorb stateŌéēs (LE_32 0)

-- All polynomial evaluations in canonical order.
-- Simple-selector fixed evals (where simpleSelectorMask[i] = True) are substituted
-- with 1 by the verifier but are NOT absorbed ŌĆö the prover omits them from the stream.
allEvals = [0]                          -- instance_poly_eval placeholder (always 0)
        ++ prfAdviceEvals               -- naq entries
        ++ fixForTranscript             -- nfq_total ŌłÆ num_simple_selectors entries
        ++ prfPermSigmaEvals            -- npc entries
        ++ prfPermProdEvals             -- num_ppe entries
        ++ prfLogupEvals                -- ╬Ż_k(nc_k + 3) entries across all lookups
        ++ prfTrashEvals                -- ntrash entries
        ++ prfDummyEvals                -- fewer-point-sets dummy evals (0 if none)
-- where fixForTranscript = [ v | (isSel, v) <- zip simpleSelectorMask prfFixedEvals, not isSel ]
stateŌéüŌéĆ = foldl (\s n -> s <> LE_32(n)) stateŌéēi allEvals

(xŌéü, stateŌéüŌéĆs) = squeeze stateŌéüŌéĆ                   // within-set combiner
(xŌéé, stateŌéüŌéĆss) = squeeze stateŌéüŌéĆs                 // across-set combiner

-- f commitment ŌåÆ xŌéā (GWC opening point)
stateŌéüŌéü = stateŌéüŌéĆss <> prfFCom
(xŌéā, stateŌéüŌéüs) = squeeze stateŌéüŌéü

-- q_evals_on_xŌéā ŌåÆ xŌéä (fold combiner)
stateŌéüŌéé = foldl (\s n -> s <> LE_32(n)) stateŌéüŌéüs prfQEvalsOnX3
(xŌéä, _) = squeeze stateŌéüŌéé
```

`LE_32(n) = integerToByteString LittleEndian 32 n`.

**Critical ordering note:** `trashChal` is squeezed once, **after** absorbing all perm
z-product commitments **and** all per-lookup LogUp commitments (helpers + accumulator),
before `prfTrashComs` are absorbed. For circuits with `ntrash = 0`, `prfTrashComs` is
empty, but the squeeze still happens at the same position.

## 5. Evaluation points

From the shared challenge `x` (evaluation point at rotation 0):

```
Žē_inv  = recip Žē                                   // ŽēŌü╗┬╣ = Žē^{N-1}
x_next = x * Žē                                      // rotation +1
x_prev = x * Žē_inv                                  // rotation ŌłÆ1
x_last = x * Žē^{-(blinding+1)}                      // last usable row
```

`Žē^{-(blinding+1)}` is computed as `(Žē_inv)^{blinding+1}` by iterated multiplication
(blinding is small ŌĆö 5 or 6 in the test circuits).

Domain size: `N = 2^k`. `x^N` is computed by k squarings via `scale N x` which uses
`expModInteger (unScalar x) N q`.

## 6. Gate constraint check

### Why it is needed

The KZG multi-point opening proof checks only that polynomial commitments are consistent with
their claimed evaluations. It does NOT verify that the vanishing quotient commitments
`prfHComs` encode the constraint polynomial `h(X) = ╬Ż constraintßĄó(X) / (X^N ŌłÆ 1)`.

A malicious prover can commit to the zero polynomial for all h-pieces, claim `h(x) = 0`, and
still produce a valid KZG opening ŌĆö because zero evaluates to zero everywhere. The gate
constraint check closes this gap by verifying:

```
hEval ├Ś (x^N ŌłÆ 1)  =  ╬ŻßĄó yŌü▒ ├Ś constraintßĄó(x)
```

where `hEval` is derived from the prover's claimed evaluations (via the H-slot assembly in
`assembleRotationSets`) and the right-hand side is the Horner-folded constraint sum.

### 6.1 Expression tree evaluator (`evalGate`)

`evalGate expr advEvals fixEvals instEvals` evaluates a `GateExpr` tree by structural
recursion.  The tree is built once at parse time by `instrsToGateExpr` in `JsonParser.hs`
(see ┬¦3.3); no stack or byte-scanning occurs at on-chain evaluation time.

```
evalGate (GEConst s)   adv fix inst = s
evalGate (GEAdv qi)    adv fix inst = adv[qi]
evalGate (GEFix qi)    adv fix inst = fix[qi]
evalGate (GEInst qi)   adv fix inst = inst[qi]
evalGate (GENeg e)     adv fix inst = 0 ŌłÆ evalGate e
evalGate (GEAdd a b)   adv fix inst = evalGate a + evalGate b
evalGate (GEMul a b)   adv fix inst = evalGate a ├Ś evalGate b
evalGate (GEScale e s) adv fix inst = evalGate e ├Ś s
```

All operand lists (`advEvals`, `fixEvals`, `instEvals`) contain values already lifted to
`Scalar`:

- `advEvals = map mkScalar prfAdviceEvals` ŌĆö `naq` elements, in `cs.advice_queries()` order
- `fixEvals = map mkScalar prfFixedEvals` ŌĆö `nfq` elements, in `cs.fixed_queries()` order
- `instEvals = [zero, pubInstEval]` ŌĆö always exactly 2 elements:
  - index 0: `zero` (hardcoded; col 0 is the zero polynomial in midnight-zk)
  - index 1: public instance column eval, computed via Lagrange interpolation (see below)

**Public instance eval:** midnight-zk places public inputs in instance column 1 (rows 0..npŌłÆ1).
The verifier interpolates:

```
nInv = recip (mkScalar N)
lagrangeAtRow i = Žē^i ├Ś (x^N ŌłÆ 1) ├Ś nInv ├Ś recip(x ŌłÆ Žē^i)
pubInstEval = ╬ŻßĄó mkScalar(pubInputs[i]) ├Ś lagrangeAtRow(i)
```

This computation is exact: the Lagrange basis polynomial `L_i(x)` equals 1 at row i and 0
at all other rows of the domain.

### 6.2 Gate expressions

```
evalE e   = evalGate e advEvals fixEvals instEvals
gateExprs = map evalE (vkGatePolys vk)
```

Each element of `vkGatePolys` is a `GateExpr` tree for one gate constraint polynomial. The
evaluation must equal zero on all active rows for a valid witness; the gate constraint check
verifies this implicitly via the Horner fold and the h-polynomial reconstruction.

### 6.3 Permutation argument constraints

Lagrange polynomials used in the permutation constraints:

```
l0    = lagrangeAtRow 0                             // L_0(x)
lLast = lagrangeAtRow (N ŌłÆ blinding ŌłÆ 1)            // L_{last}(x)
lBlind = ╬ŻŌ▒╝ŌéīŌéĆ^{blindingŌłÆ1} lagrangeAtRow (N ŌłÆ blinding + j)
activeRows = 1 ŌłÆ lLast ŌłÆ lBlind
```

For each permutation chunk j (0..npŌłÆ1), where `np = ceil(npc / chunk_size)`:

```
-- Evaluations (from prfPermProdEvals via the flat layout in ┬¦3.5.1):
zj_x    = mkScalar prfPermProdEvals[3j]
zj_xw   = mkScalar prfPermProdEvals[3j+1]
zj_xlast = mkScalar prfPermProdEvals[3j+2]  // only for non-last chunk

-- Coset shifts and column evaluations for columns in chunk j
-- (columns j*chunk_size .. min((j+1)*chunk_size, npc) ŌłÆ 1):
For each column k in chunk j (local_k = 0, 1, ...):
  global_k = j * chunk_size + local_k
  (col_type, eval_idx) = vkPermColTypes[global_k]
  col_eval = advEvals[eval_idx] | fixEvals[eval_idx] | instEvals[eval_idx]
  sigma_eval = mkScalar prfPermSigmaEvals[global_k]
  coset_shift = ╬┤^global_k        // bls12_381_scalar_delta ^ global_k (see ┬¦2)
```

Four constraint expressions per chunk:

```
-- (1) Initialisation: z_j(Žē^0 ┬Ę x) = 1 at row 0
e_init = l0 ├Ś (1 ŌłÆ zj_x)

-- (2) Finalisation: z_j(x) Ōłł {0,1} at last row
e_final = lLast ├Ś (zj_x ├Ś zj_x ŌłÆ zj_x)

-- (3) Product argument (on active rows):
lhs = zj_xw ├Ś ╬ĀŌé¢ (col_eval_k + ╬▓ ├Ś sigma_eval_k + ╬│)
rhs = zj_x  ├Ś ╬ĀŌé¢ (col_eval_k + ╬▓ ├Ś coset_shift_k ├Ś x + ╬│)
e_prod = activeRows ├Ś (lhs ŌłÆ rhs)

-- (4) Linking between chunks (only for j < npŌłÆ1):
z_next_xlast = mkScalar prfPermProdEvals[3(j+1)+2]  // next chunk at x┬ĘŽē^last
e_link = l0 ├Ś (z_{j+1}(x) ŌłÆ zj_xlast)
```

The Horner fold processes them in this global order (not per-chunk):

```
[1]           lŌéĆ ├Ś (1 ŌłÆ Z_0(x))                            ŌåÉ init (chunk 0 only)
[2]           l_last ├Ś (Z_{npŌłÆ1}(x)┬▓ ŌłÆ Z_{npŌłÆ1}(x))        ŌåÉ final (last chunk only)
[3..np+1]     lŌéĆ ├Ś (Z_{j+1}(x) ŌłÆ Z_j(x┬ĘŽē^last))  j=0..npŌłÆ2  ŌåÉ chunk links
[np+2..2np+1] activeRows ├Ś (LHS_j ŌłÆ RHS_j)          j=0..npŌłÆ1  ŌåÉ product constraints
```

Total permutation expressions: `2 ├Ś np + 1 = 2 ├Ś numChunks + 1`.

### 6.4 Lookup argument constraints

midnight-zk uses the **LogUp** lookup argument (v7), which replaces the old Plookup
grand-product scheme with a logarithmic-derivative identity. Each lookup k has `nc_k` chunks
(one chunk groups parallel inputs whose combined degree stays within `cs_degree`), and
contributes `nc_k + 2` constraint expressions.

Let `o_k = ╬Ż_{i<k}(nc_i + 3)` be the flat offset into `prfLogupEvals` for lookup k.

```
-- Evaluations from prfLogupEvals at offset o_k:
multEval      = mkScalar prfLogupEvals[o_k]              // m_k(x) ŌĆö multiplicity poly
helperEvals   = [ mkScalar prfLogupEvals[o_k + 1 + c]    // h_{k,c}(x) for c=0..nc_k-1
                  | c ŌåÉ 0..nc_k-1 ]
accumEval     = mkScalar prfLogupEvals[o_k + nc_k + 1]   // Z_k(x)
accumNextEval = mkScalar prfLogupEvals[o_k + nc_k + 2]   // Z_k(x┬ĘŽē)

-- ╬Ė-compression of table expression (vkLookupTableExprs[k]):
t╠ä = foldl (\acc e -> acc ├Ś ╬Ė + evalE e) 0 (vkLookupTableExprs[k])

-- Lookup selector evaluation:
sel = evalE (vkLookupSelectorExprs[k])

-- For chunk c (0..nc_k-1), compress each parallel input j:
--   vkLookupInputExprs[k][c] is a list of width-expr lists, one per parallel input j.
f╠ä_{c,j} = foldl (\acc e -> acc ├Ś ╬Ė + evalE e) 0 (vkLookupInputExprs[k][c][j])
fsBeta_c = [ f╠ä_{c,j} + ╬▓  |  j ŌåÉ 0..J_c-1 ]   // J_c = number of parallel inputs in chunk c
```

The `nc_k + 2` constraint expressions, in Horner-fold order:

```
-- (1) Boundary (1 expression):
e_boundary = (lŌéĆ + l_last) ├Ś accumEval

-- (2) Helper constraint for chunk c (nc_k expressions, c = 0..nc_k-1):
--     h_c ├Ś ╬Ā_j(f╠ä_{c,j}+╬▓) ŌłÆ ╬Ż_j ╬Ā_{mŌēĀj}(f╠ä_{c,m}+╬▓) = 0
--     (first term: product of all fsBeta; second: sum of partial products)
product_c  = foldl (├Ś) 1 fsBeta_c
sumParts_c = ╬Ż_j  ╬Ā_{mŌēĀj} fsBeta_c[m]          // partial products (no field inversions)
e_helper_c = helperEvals[c] ├Ś product_c ŌłÆ sumParts_c

-- (3) Accumulator constraint (1 expression):
sumH = ╬Ż_c helperEvals[c]
e_accum = activeRows ├Ś ((accumNextEval ŌłÆ accumEval ŌłÆ sel ├Ś sumH) ├Ś (t╠ä + ╬▓) + multEval)
```

**Note on ╬▓ and ╬│**: LogUp uses only **╬▓** for the lookup identity. The challenge **╬│** is
used exclusively in the permutation argument (┬¦6.3). This distinguishes LogUp from the
old Plookup protocol which required both ╬▓ and ╬│ for lookups.

### 6.5 Trash argument constraints

For each trashcan t (0..ntrashŌłÆ1), where `ntrash = len(vkTrashSelectors vk)`:

```
trash_eval = mkScalar prfTrashEvals[t]
sel = evalE (vkTrashSelectors vk !! t)

-- trashChal-compression of constraint expressions:
compressed = foldl (\acc e -> acc ├Ś trashChal + evalE e) 0 (vkTrashConstraintExprs vk !! t)

e_t = compressed ŌłÆ (1 ŌłÆ sel) ├Ś trash_eval
```

### 6.6 Horner folding and equality check

All constraint expressions are concatenated in order:

```
allExprs = gateExprs ++ permExprs ++ lookupExprs ++ trashExprs
```

where:
- `gateExprs`: one eval per gate polynomial (length G)
- `permExprs`: 1 init + 1 final + (npŌłÆ1) links + np products = 2├Śnp+1 total (┬¦6.3 order)
- `lookupExprs`: for each lookup k: [e_boundary, e_helper_0, ŌĆ”, e_helper_{nc_kŌłÆ1}, e_accum]
  (nc_k + 2 per lookup; total nL = ╬Ż_k(nc_k + 2))
- `trashExprs`: one expr per trashcan (ntrash total)

Horner fold with challenge y:

```
hEvalSum = foldl (\acc e -> acc ├Ś y + e) 0 allExprs
```

Equality check (the only circuit-level soundness assertion):

```
hEval ├Ś xnMinusOne  ==  hEvalSum
```

where `hEval` is the value assembled for the H slot in `assembleRotationSets` (see ┬¦7.2)
and `xnMinusOne = x^N ŌłÆ 1`.

## 7. Rotation set assembly

`assembleRotationSets vk prf specs x xŌéü x_next x_prev x_last hEval`

For each `RotationSetSpec` in `specs`:

1. Convert each raw integer rotation offset to a `Rotation` ADT value via `toRotation`
   (conversion happens here because `ccBlinding` is required to identify `RotLast`):

   | Integer offset | `Rotation` | Evaluation point |
   |----------------|-----------|-----------------|
   | `0` | `RotCur` | `x` |
   | `+1` | `RotNext` | `x_next` |
   | `-1` | `RotPrev` | `x_prev` |
   | `-(blinding+1)` | `RotLast` | `x_last` |
   | any other | `RotArb r` | `x ┬Ę Žē^r` (iterated multiply) |

   `RotArb` does not appear in midnight-zk circuits; it is kept for completeness.

2. For each slot (in xŌéü-power order), with accumulated power `xŌéü^j`:
   - Look up the commitment and evaluation(s) using the poly-kind dispatch below.
   - Add `xŌéü^j ├Ś commitment` to the set's combined commitment (`rsScaledComs`).
   - For each rotation point `pts[r]`, add `xŌéü^j ├Ś eval_at_rotation_r` to `rsQEvalsAtPts[r]`.

3. Return `RotationSet { rsPoints, rsScaledComs, rsQEvalsAtPts }` where `rsPoints` are the
   resolved evaluation points.

The H slot (kind `"H"`) is the only slot that expands to multiple physical commitments (see ┬¦7.2).

### 7.1 Poly-kind dispatch table

| Kind | Commitment | Eval at rotation |
|------|-----------|------------------|
| `Advice` | `uncompress prfAdviceComs[ssIndex]` | `unifiedEvals[ssEvalIdxs[rotPos]]` |
| `Instance` | `G1_zero` | `zero` (hardcoded ŌĆö col 0 is zero poly) |
| `LogupMult` | `uncompress prfLookupMultComs[ssIndex]` | `unifiedEvals[ssEvalIdxs[rotPos]]` |
| `Trash` | `uncompress prfTrashComs[ssIndex]` | `unifiedEvals[ssEvalIdxs[rotPos]]` |
| `Fixed` | `uncompress vkFixedComs[ssIndex]` | `unifiedEvals[ssEvalIdxs[rotPos]]` |
| `PermSigma` | `uncompress vkPermSigmaComs[ssIndex]` | `unifiedEvals[ssEvalIdxs[rotPos]]` |
| `H` | (see ┬¦7.2) | `linComEval` at RotCur; `unifiedEvals[ssEvalIdxs[rotPos]]` otherwise |
| `PermProd` | `uncompress prfPermProdComs[ssIndex]` | `unifiedEvals[ssEvalIdxs[rotPos]]` |
| `LogupAccum` | `uncompress prfLookupAccumComs[ssIndex]` | `unifiedEvals[ssEvalIdxs[rotPos]]` |
| `LogupHelper` | `uncompress prfLookupHelperComs[ssIndex]` (flat concat across lookups) | `unifiedEvals[ssEvalIdxs[rotPos]]` |

`uncompress` = `bls12_381_G1_uncompress`.

`unifiedEvals` is the concatenation built in `assembleRotationSets`:
```
unifiedEvals = advEvalsRS | fixEvalsRS | permSigEvalsRS | permProdEvalsRS |
               logupEvalsRS | trashEvalsRS | dummyEvalsRS
```
All `eval_idxs` in `*_rotation_sets.json` are absolute offsets into this array.

`LogupHelper` uses a flat index `ssIndex` into the concatenation of all helper commitment lists
across all lookups: `prfLookupHelperComs = concat [helpers_0, helpers_1, ŌĆ”]`.

### 7.2 H-piece special case: the linearization commitment

midnight-zk does not commit to `h(X)` directly. The prover commits to the **linearization
polynomial**:

```
L(X) = (1 ŌłÆ x^n) ┬Ę h(X) + ╬Ż_k c_k ┬Ę S_k(X)
```

where `h(X) = hŌéĆ(X) + X^{NŌłÆ1}┬ĘhŌéü(X) + ŌĆ”` is the vanishing quotient, `S_k` are the
**simple-selector** fixed columns (boolean columns that are not sent in the proof stream),
and `c_k` are Fiat-Shamir-derived scalars computed by `computeHEval` (the `selColData` output).

#### Why L(X) rather than h(X)?

Simple-selector columns are optimised away from the multi-open proof stream by midnight-proofs:
the prover knows `S_k(x) = 1` at any random x (they are boolean with a sparse support that
doesn't include random field elements), so the verifier can substitute 1 without receiving
those evaluations. But the commitment `[S_k(s)]Ōéü` must still appear somewhere to bind the
prover. midnight-proofs folds the `S_k` commitments into `L(X)` with coefficients `c_k`, so
the H slot commitment is a linear combination of `[h_j(s)]Ōéü` and `[S_k(s)]Ōéü` pieces.

#### Evaluation at x

At x, substituting `S_k(x) = 1` (simple-selector substitution):

```
L(x) = (1 ŌłÆ x^n) ┬Ę h(x) + ╬Ż_k c_k
      = ŌłÆ(x^n ŌłÆ 1) ┬Ę hEval + ╬Ż_k c_k
      =: linComEval
```

This is what `computeHEval` returns as `linComEval`. The H slot at `RotCur` uses
`linComEval` as its evaluation (not `hEval` directly, and not from the proof stream).

#### Commitment MSM for the H slot

The H slot (logical index `hSlotPos` in the rotation set, with accumulated power `xŌéü^{hSlotPos}`)
expands to `nh + |selCols|` physical G1 points in the MSM:

```
-- h-piece contributions (l = 0..nhŌłÆ1):
scalar_l = xŌéü^{hSlotPos} ├Ś (1 ŌłÆ x^n) ├Ś hSplit^l    // hSplit = x^{N-1}
point_l  = uncompress prfHComs[l]

-- simple-selector column contributions (k = 0..|selColData|ŌłÆ1):
scalar_k = xŌéü^{hSlotPos} ├Ś c_k                      // c_k from selColData
point_k  = uncompress vkFixedComs[colIdx_k]          // colIdx_k from selColData
```

`powers hSplit nh` produces `[1, hSplit, hSplit┬▓, ŌĆ”, hSplit^{nhŌłÆ1}]`.

#### At other rotations (fewer-point-sets dummy evals)

When the `fewer-point-sets` feature is active, the H slot may appear in a multi-rotation set
(e.g. `{x, xŽē}`). At non-current rotations the H slot's eval comes from the dummy
evaluations appended to `prfDummyEvals` ŌĆö `unifiedEvals[ssEvalIdxs[rotPos]]` in the unified
eval array (see ┬¦3.5 and ┬¦3.6). These dummy evals are synthetic values the prover generates
so the KZG opening remains consistent across the merged set.

#### Soundness

`computeHEval` derives `hEval = hEvalSum / (x^nŌłÆ1)` from the Horner-folded constraint sum.
It then computes `linComEval` from `hEval` and `selColData`. The KZG multi-point opening
check in `verifyGwc` enforces that the committed `L(X)` polynomial opens to exactly
`linComEval`. A prover who committed to a fraudulent `L` (not encoding the correct `h` and
selector columns) would produce a commitment inconsistent with `linComEval`, causing the
final pairing check to fail.

## 8. Generic GWC multi-point opening

`verifyGwc sG2 fCom xŌéé xŌéā xŌéä rotSets qEvalsOnX3 piPt`

Given the assembled `[RotationSet]` and the challenges from the transcript:

**Step 1: Lagrange interpolants**

For each rotation set i, compute `r_i(xŌéā)` ŌĆö the unique polynomial of degree < |SßĄó| that
passes through the points `{(rsPoints[k], rsQEvalsAtPts[k])}`:

```
r_i(xŌéā) = lagrange rsPoints rsQEvalsAtPts xŌéā
```

Lagrange interpolation for `m` points:

```
lagrange pts evals z = ╬ŻŌé¢ evals[k] ├Ś ╬ĀŌ▒╝ŌēĀŌé¢ (z ŌłÆ pts[j]) / (pts[k] ŌłÆ pts[j])
```

**Step 2: Vanishing polynomial evaluations**

```
V_i(xŌéā) = ╬ĀŌé¢ (xŌéā ŌłÆ rsPoints[k])
```

**Step 3: GWC contributions**

```
c_i = (qEvalsOnX3[i] ŌłÆ r_i(xŌéā)) ├Ś recip(V_i(xŌéā))
```

**Step 4: f(xŌéā)**

```
f(xŌéā) = ╬ŻßĄó xŌéé^i ├Ś c_i      (Horner: horner xŌéé cs)
```

**Step 5: Combined commitment and evaluation**

```
finalCom = ╬ŻßĄó xŌéä^i ├Ś qCom_i  +  xŌéä^m ├Ś fCom
vEval    = ╬ŻßĄó xŌéä^i ├Ś qEvalsOnX3[i]  +  xŌéä^m ├Ś f(xŌéā)
```

where `qCom_i = ╬ŻŌ▒╝ rsScaledComs_i[j]` (the summed pre-scaled commitments for set i).

In Plutus these MSMs use `bls12_381_G1_multiScalarMul`.

**Step 6: KZG pairing** (see ┬¦9)

## 9. KZG pairing check

The KZG opening relation `f(xŌéā) = eval` for commitment `[f]Ōéü = finalCom` with witness
`ŽĆ = piPt`:

```
e(piPt, sG2) = e(finalCom ŌłÆ vEval┬ĘG1_gen + xŌéā┬ĘpiPt, G2_gen)
```

In Plutus:

```
ml1 = bls12_381_millerLoop piPt sG2
ml2 = bls12_381_millerLoop (finalCom - vEval┬ĘG1_gen + xŌéā┬ĘpiPt) G2_gen
bls12_381_finalVerify ml1 ml2
```

The right-hand G2 argument is the **positive** generator `bls12_381_G2_compressed_generator`
(decompressed). Using its negation would give False.

## 10. Circuit-specific parameters

All 10 test circuits use `cs_degree = 5` (so `chunk_size = 3`, `nh = 4`).
`blinding_factors` is 5 for four circuits and 6 for the other six ŌĆö see the table below.

`ntrash` (`num_blinding_commitments` in circuit_params.json) is the count of extra G1
blinding commitment points in the proof. It is derived from the actual proof byte length
rather than from a fixed formula, so circuits with the same `blinding_factors` can have
different `ntrash` values (e.g. Ethereum ECDSA has `blinding_factors = 6` but `ntrash = 0`).

| Bitcoin Schnorr | 5 | 9 | 7 | 12 | 4 | 25 | 42 | 0 | 11 | 5 |
| Poseidon hash | 6 | 8 | 1 | 8 | 3 | 11 | 19 | 1 | 8 | 4 |
| JubJub ECC | 6 | 9 | 1 | 10 | 4 | 12 | 22 | 1 | 11 | 4 |
| ECDSA threshold | 5 | 9 | 5 | 12 | 4 | 24 | 27 | 0 | 11 | 5 |

# midnight-zk Generic Halo2 GWC Verifier — Specification

This document is the authoritative specification for the generic midnight-zk Halo2 GWC
verifier implemented in `plutus-midnight-zk/`. It is intended to be self-contained: an
auditor should be able to read this document and independently verify every step of the
implementation in `Verifier.hs` and `JsonParser.hs` against the JSON test vectors.

## 1. Architecture

```
*_plutus_vk.json           ─┐
*_circuit_constraint.json  ─┴→ parsePlutusVK    → VerifyingKey (incl. CircuitConfig)
*_plutus_proof.json          → parsePlutusProof → Proof
*_rotation_sets.json         → parseRotationSets → [RotationSetSpec]
*_plutus_instance.json       → parseInstance  → [Integer]

verify vk proof specs pubInputs
  ├─ Transcript → x, x₁, x₂, x₃, x₄, y, θ, β, γ, trashChal
  ├─ computeHEval → h(x)          (gate constraint soundness check)
  ├─ assembleRotationSets → [RotationSet]  (circuit-aware, spec-driven)
  └─ verifyGwc → Bool             (fully circuit-agnostic KZG check)
        └─ bls12_381_finalVerify  (single pairing equation)
```

The Rust binary `write-test-vectors` produces all JSON artifacts by running the
midnight-zk circuits through their prove/verify cycle and serialising the relevant
data structures. The Haskell verifier reads these artifacts and re-runs only the
verification steps.

## 2. Notation and field arithmetic

| Symbol | Meaning |
|--------|---------|
| `F_q` | BLS12-381 scalar field, prime `q = 52435875175126190479447740508185965837690552500527637822603658699938581184513` |
| `F_p` | BLS12-381 base field (used for G1/G2 coordinates) |
| `G1`, `G2` | BLS12-381 groups; additive notation |
| `G1_0` | G1 identity (point at infinity); compressed form = 48 zero bytes with high bit set |
| `[a]₁` | `a · G1_gen` |
| `[s]₂` | `s · G2_gen` (from SRS) |
| `e(P, Q)` | Optimal Ate pairing; `bls12_381_millerLoop` + `bls12_381_finalVerify` in Plutus |
| `LE(bs)` | Interpret byte string `bs` as a little-endian non-negative integer |
| `msm(ss, Ps)` | Multi-scalar multiplication `Σᵢ sᵢ · Pᵢ`; `bls12_381_G1_multiScalarMul` |
| `blake2b_256(bs)` | 32-byte BLAKE2b-256 hash of `bs` |
| `Scalar n` | Field element `n mod q` |
| `δ` | BLS12-381 scalar field coset generator: `δ = 7^{2^32} mod q = 3793952369011177517951424454785176000433849974408744014172535497121832470999`. Hardcoded as `bls12_381_scalar_delta` in `BlsUtils.hs`. See §6.3. |

**Field structure note (why δ works as a coset generator):**
`F_q*` is cyclic of order `q − 1 = T · 2^32` with `T` odd, generated by `7`.
Since `gcd(T, 2^32) = 1`, the Chinese Remainder Theorem gives an internal direct product:
```
F_q* ≅ ⟨ω⟩ × ⟨δ⟩
```
where `ω = 7^T` has order `2^32` and `δ = 7^{2^32}` has order `T`.
The two subgroups share only the identity `{1}`.

Let `H = {1, ω^{2^32/n}, ω^{2·2^32/n}, …}` be the evaluation domain of size `n = 2^k`.
The cosets `δ^0·H, δ^1·H, δ^2·H, …` are pairwise disjoint: if `δ^i·h₁ = δ^j·h₂` for
`h₁,h₂ ∈ H` then `δ^{i−j} = h₂·h₁⁻¹ ∈ H`, but `δ^{i−j}` has odd order and every
element of `H` has 2-power order, so both sides must be `1`, giving `i = j`.
The permutation argument encodes cell `(column i, row j)` with the unique identity
`δ^i · ω^j`, so a prover cannot conflate values across different columns.

All hex strings in the JSON are **little-endian**: byte 0 is the least significant byte.

Field inversion uses the new modular exponentiation plutus builtin via
`expModInteger a (-1) q` in Plutus.

## 3. JSON test vector formats

For human readability all artifacts are written to JSON, as it is more transparent than raw bytes.

### 3.1 Plutus VK JSON

File: `*_plutus_vk.json`. Produced by `write_json_all_artifacts` in `circuit_params.rs`.

Contains the **trusted-setup-dependent** fields only: the polynomial commitments, SRS G2
point, and the domain parameters derived from the circuit's final compiled form.  These
fields change when you redo the trusted setup (new SRS), but are otherwise fixed for a
given circuit.

All G1 commitments are 48-byte compressed BLS12-381 points encoded as hex strings.
All scalars are 32-byte little-endian field elements encoded as hex strings.

```json
{
  "k":                     <integer>,        // log₂(domain size); N = 2^k
  "omega":                 "<32-byte LE hex>", // primitive N-th root of unity ω
  "blinding_factors":      <integer>,        // number of blinding rows (5 or 6 in the test circuits)
  "num_advice_columns":    <integer>,        // na
  "num_perm_columns":      <integer>,        // npc (= len(permutation_commitments))
  "cs_degree":             <integer>,        // constraint system degree d; chunk_size = d-2, nh = d-1
  "num_lookups":           <integer>,        // nl
  "fixed_commitments":     ["<48-byte hex>", ...],            // one per fixed column
  "permutation_commitments": ["<48-byte hex>", ...],          // npc entries (sigma polys)
  "s_g2":                  "<96-byte hex>",  // [s]G₂ from SRS
  "transcript_repr":       "<32-byte hex>"   // VK identity hash (Fiat-Shamir seed)
}
```

**What changes with a new SRS**: `fixed_commitments`, `permutation_commitments`, `s_g2`,
and `transcript_repr` are all derived from the SRS randomness.  `k`, `omega`,
`blinding_factors`, and the scalar dimensions are fixed by the circuit design.

**Why `advice_queries` / `fixed_queries` are absent**: These arrays are committed to via
`transcript_repr` on the Rust side, so they are implicitly bound to the Fiat-Shamir
challenges.  The verifier never needs to read them directly: evaluation indices are
pre-computed into `eval_idxs` in `*_rotation_sets.json`, and `naq` / `nfq` are available
in `*_circuit_params.json`.

### 3.2 Circuit constraint JSON

File: `*_circuit_constraint.json`. Produced by `write_json_all_artifacts` in `circuit_params.rs`.

Contains the **circuit-design-dependent** fields: the symbolic polynomial expressions
that encode the constraint system.  These change only when the circuit logic changes,
not when the trusted setup (SRS) is redone.  Analogous to `*_rotation_sets.json`, which
is also circuit-design-dependent and in its own file.

```json
{
  "gate_polys":            [ <instr_array>, ... ],  // one RPN instruction array per gate poly
  "perm_col_types":        [{"col_type":<0|1|2>,"eval_idx":<int>}, ...],  // npc entries
  "lookup_input_exprs":    [ [<instr_array>, ...], ... ],  // nl × (variable number) of expressions
  "lookup_table_exprs":    [ [<instr_array>, ...], ... ],  // nl × (variable number) of expressions
  "trash_selectors":       [ <instr_array>, ... ],  // one per trashcan argument
  "trash_constraint_exprs": [ [<instr_array>, ...], ... ]  // one list of exprs per trashcan
}
```

**`perm_col_types`** encodes how to look up each permutation column's evaluation at `x`:

| `col_type` | Column type | Evaluation source |
|------------|-------------|-------------------|
| `0` | Advice | `prfAdviceEvals[eval_idx]` |
| `1` | Fixed | `prfFixedEvals[eval_idx]` |
| `2` | Instance | `instEvals[eval_idx]` |

`eval_idx` is the position in the corresponding evaluation list at `Rotation::cur()` (i.e.,
the query at rotation 0 for that column), as it appears in `cs.advice_queries()` /
`cs.fixed_queries()` / `cs.instance_queries()`.

**Note on δ (delta):** The coset generator δ used in the permutation argument (see §6.3) is
**not** stored in this file.  It is a pure constant of the BLS12-381 scalar field —
`δ = 7^{2^32} mod q` — and is hardcoded as `bls12_381_scalar_delta` in `BlsUtils.hs`.

All instruction arrays (`gate_polys`, `lookup_*_exprs`, `trash_*`) use the RPN format
described in §3.3.

### 3.3 Gate polynomial RPN instruction format

All symbolic polynomial fields in `*_circuit_constraint.json` — `gate_polys`,
`lookup_input_exprs`, `lookup_table_exprs`, `trash_selectors`, `trash_constraint_exprs` —
are stored as **flat RPN instruction arrays** — sequences of JSON objects, each with an
`"op"` field, encoding a polynomial expression in postfix order.

`instrsToGateExpr` in `JsonParser.hs` converts each array into a `GateExpr` recursive
expression tree (see §6.1) by simulating the RPN stack at parse time with `GateExpr` nodes
as stack values.  Constants and scale factors are decoded from hex at parse time and stored
directly in the tree — no byte-scanning occurs at on-chain evaluation to save on the plutus budget.

#### Instruction objects

| JSON object | `GateExpr` constructor |
|-------------|------------------------|
| `{"op":"Constant","value":"<32-byte LE hex>"}` | `GEConst (decoded Scalar)` |
| `{"op":"Advice","query_index":<uint>}` | `GEAdv qi` |
| `{"op":"Fixed","query_index":<uint>}` | `GEFix qi` |
| `{"op":"Instance","query_index":<uint>}` | `GEInst qi` |
| `{"op":"Negated"}` | `GENeg a` |
| `{"op":"Sum"}` | `GEAdd a b` |
| `{"op":"Product"}` | `GEMul a b` |
| `{"op":"Scaled","factor":"<32-byte LE hex>"}` | `GEScale a (decoded Scalar)` |

`query_index` is an index into the corresponding evaluation list, in the order those queries
appear in the circuit's `advice_queries()` / `fixed_queries()` / `instance_queries()`.

In Halo2, `Selector` is a boolean-like column used to conditionally enable gates on specific rows.
During `keygen_vk`, the compiler multiplies each selector value into the gate polynomial expressions
as a fixed column, eliminating `Selector` nodes from the symbolic representation entirely. By the
time the verifying key is finalised, only `Advice`, `Fixed`, `Instance`, `Constant`, `Sum`,
`Product`, `Negated`, and `Scaled` nodes remain. A `Selector` instruction in the JSON would
indicate that the upstream Rust compilation pipeline failed to compile it away — a bug in circuit
key generation, not something `instrsToGateExpr` is expected to handle.

midnight-zk circuits have no multi-phase challenges (though it can support it), so the `Challenge` instruction will
not appear in practice and is not handled by `instrsToGateExpr`. E.g, this is a single round
verifier, the prover sends only one message to the verifier (the smart contract).

#### Example

The expression `(Fixed[9] + Advice[0]) * Constant[c] − Advice[1]` encodes as:

```json
[
  {"op": "Fixed",    "query_index": 9},
  {"op": "Advice",   "query_index": 0},
  {"op": "Sum"},
  {"op": "Constant", "value": "..."},
  {"op": "Product"},
  {"op": "Advice",   "query_index": 1},
  {"op": "Negated"},
  {"op": "Sum"}
]
```

`instrsToGateExpr` builds the tree
`GEAdd (GENeg (GEAdv 1)) (GEMul (GEConst c) (GEAdd (GEFix 9) (GEAdv 0)))`
reading from the stack state after processing each instruction in postfix order.

> **Why an ADT instead of keeping RPN?**
> Haskell's algebraic data types enable structural recursion (a simple `go` catamorphism
> in `evalGate`) rather than an explicit stack machine. This eliminates the partial error
> cases that a runtime stack needs (underflow, leftover values), makes the evaluator a
> total function, and plays well with PlutusTx serialisation via `makeIsDataIndexed`.
> The RPN-to-tree conversion is paid once at parse time inside `instrsToGateExpr`; the
> on-chain evaluator never sees raw instruction arrays.

### 3.4 Circuit params JSON

File: `*_circuit_params.json`. Produced by `write_json_circuit_artifacts`.

> **Note:** this file is not read by the Haskell parser or verifier. All dimensions it lists
> are either already present in the other JSON artifacts or derived at runtime from array
> lengths. Its purpose is human-readable reference: a compact snapshot of circuit dimensions
> useful for auditing, comparing circuits side-by-side, or benchmarking (e.g. tracking how
> `num_advice_columns` or `num_rotation_sets` change across circuit versions).

```json
{
  "num_advice_columns":           <integer>,  // na — advice (witness) columns
  "num_lookups":                  <integer>,  // nl — lookup arguments
  "num_permutation_commitments":  <integer>,  // npc — columns covered by permutation
  "permutation_chunk_size":       <integer>,  // chunk_size = cs_degree − 2
  "num_quotient_commitments":     <integer>,  // nh = cs_degree − 1
  "num_advice_queries":           <integer>,  // naq — total advice evaluations in proof
  "num_fixed_queries":            <integer>,  // nfq — total fixed evaluations in proof
  "num_blinding_commitments":     <integer>,  // ntrash — extra G1 blinding points in the proof (0 or 1)
  "num_permutation_product_evals":<integer>,  // nppe = (np−1)×3 + 2
  "num_rotation_sets":            <integer>   // nq — must equal num_sets in rotation_sets.json
}
```

**`blinding_factors`** (in the VK) is the number of blinding rows appended to each column
in the witness. It controls the last-usable-row offset (`x_last = x · ω^{-(blinding+1)}`)
and the Lagrange polynomials `lLast` and `lBlind` in the constraint check. The test circuits
use `blinding_factors` of 5 or 6 (see §10).

**`num_blinding_commitments`** (in circuit_params.json) is a separate quantity: the count of
extra G1 blinding commitment points (`prfTrashComs`) appended to the proof transcript. It is
determined from the actual proof size, not directly from `blinding_factors`. A circuit with
`blinding_factors = 6` may still have `num_blinding_commitments = 0` (e.g. Ethereum ECDSA).

### 3.5 Rotation sets JSON

File: `*_rotation_sets.json`. Produced by `rotation_sets_bytes()` in `rotation_sets.rs`.

```json
{
  "num_sets": <integer>,
  "sets": [
    {
      "rotations": [<i32>, ...],       // rotation offsets: 0, ±1, -(blinding+1)
      "slots": [
        {
          "poly_kind": "<string>",     // see poly-kind table below
          "index":     <integer>,      // primary index (meaning depends on poly_kind)
          "eval_idxs": [<integer>, ...]  // one per rotation; used by Advice and Fixed only
        },
        ...
      ]
    },
    ...
  ]
}
```

Sets are in GWC sorted order: ascending cardinality, ties broken by first-encounter index.
Slots within a set are in x₁-power order (the order they are multiplied by powers of x₁).

#### Poly-kind table

| `poly_kind` | `index` meaning | Commitment | Evaluation |
|-------------|-----------------|-----------|------------|
| `"Advice"` | advice column index | `prfAdviceComs[index]` | `prfAdviceEvals[eval_idxs[rotPos]]` |
| `"Instance"` | (ignored — see note) | G1 identity | `zero` (hardcoded) |
| `"LookupTable"` | lookup index k | `prfLookupTableComs[k]` | `prfLookupEvals[5k+4]` |
| `"Trash"` | trash index k | `prfTrashComs[k]` | `prfTrashEvals[k]` |
| `"Fixed"` | fixed column index | `vkFixedComs[index]` | `prfFixedEvals[eval_idxs[rotPos]]` |
| `"PermSigma"` | sigma column index | `vkPermSigmaComs[index]` | `prfPermSigmaEvals[index]` |
| `"H"` | (ignored) | all `prfHComs[0..nh−1]` with hSplit scaling | `hEval` (computed by verifier) |
| `"Random"` | (ignored) | `prfRandomCom` | `prfRandomEval` |
| `"PermProd"` | chunk index j | `prfPermProdComs[j]` | `prfPermProdEvals[...]` (see §3.5.1) |
| `"LookupProd"` | lookup index k | `prfLookupProdComs[k]` | `prfLookupEvals[5k+0]` (rot 0) or `[5k+1]` (rot +1) |
| `"LookupInput"` | lookup index k | `prfLookupInputComs[k]` | `prfLookupEvals[5k+2]` (rot 0) or `[5k+3]` (rot −1) |

> **Why `index` is ignored for `Instance`, `H`, and `Random`:**
> The GWC KZG opening is fully generic — it processes every slot as a `(commitment, evaluation)`
> pair regardless of column type. Three kinds deviate from the normal "use `index` to look up
> which polynomial" pattern for Halo2-specific reasons:
>
> - **`Instance`**: midnight-zk never commits to an instance column in the normal sense.
>   Instance column 0 is the zero polynomial (public inputs are handled via Lagrange
>   interpolation on a separate column), so its commitment is always the G1 identity and its
>   evaluation is always 0. Both values are fixed by convention; nothing is read from the proof
>   and `index` is meaningless.
>
> - **`H`** (quotient polynomial): `h(X)` is split into `nh` physical pieces
>   `prfHComs[0..nh−1]`, all of which are used together scaled by powers of
>   `hSplit = x^{N-1}` — there is no single indexed commitment to pick. Its evaluation
>   `hEval` is not read from the proof at all; the verifier derives it from the gate
>   constraint sum (`computeHEval`) and the KZG opening then enforces consistency. So both
>   the commitment and the evaluation bypass the `index` field entirely.
>
> - **`Random`**: there is exactly one blinding polynomial per proof — a singleton, not an
>   array. `prfRandomCom` and `prfRandomEval` are read directly; there is nothing to index into.

For `"Advice"` and `"Fixed"`: `eval_idxs[rotPos]` gives the index into `prfAdviceEvals` /
`prfFixedEvals` for the rotation at position `rotPos` in the set's rotation list.

For all other kinds, `eval_idxs` contains zeros and is not used by the verifier.

#### 3.5.1 PermProd eval dispatch

`prfPermProdEvals` is a flat list of length `(np−1) × 3 + 2`:

- Non-last chunk `j` (0..np−2): indices `3j`, `3j+1`, `3j+2` = evaluations at `x`, `x·ω`, `x·ω^last`
- Last chunk (index np−1): indices `3(np−1)`, `3(np−1)+1` = evaluations at `x`, `x·ω` only

Dispatch in kind `"PermProd"`:
- rotation 0 → field 0 (index `3j`)
- rotation +1 → field 1 (index `3j+1`)
- any other (x·ω^last) → field 2 (index `3j+2`)

### 3.6 Proof JSON

File: `*_plutus_proof.json`. Produced by `proof_to_json` in `circuit_params.rs`.

All G1 points are 48-byte hex strings. All scalar evaluations are 32-byte LE hex strings.

```json
{
  "advice_commitments":              ["<48-byte hex>", ...],  // na entries [a_i]₁
  "lookup_permuted_commitments":     [
    {"permuted_input": "<hex>", "permuted_table": "<hex>"},   // nl entries: [A'_k]₁, [S'_k]₁
    ...
  ],
  "permutation_product_commitments": ["<48-byte hex>", ...],  // np entries [Z_j]₁
  "lookup_product_commitments":      ["<48-byte hex>", ...],  // nl entries [Z^lp_k]₁
  "trash_commitments":               ["<48-byte hex>", ...],  // ntrash entries
  "random_poly_commitment":          "<48-byte hex>",         // [r]₁
  "h_commitments":                   ["<48-byte hex>", ...],  // nh entries [h_j]₁

  "advice_evals":                    ["<32-byte hex>", ...],  // naq entries, in cs.advice_queries() order
  "fixed_evals":                     ["<32-byte hex>", ...],  // nfq entries, in cs.fixed_queries() order
  "random_eval":                     "<32-byte hex>",         // r(x)
  "sigma_evals":                     ["<32-byte hex>", ...],  // npc entries σ_i(x)
  "permutation_product_evals":       [                        // np objects
    {"eval": "<hex>", "next_eval": "<hex>", "last_eval": "<hex>" | null},
    ...
  ],
  "lookup_evals":                    [                        // nl objects, 5 evals each
    {
      "product_eval":            "<32-byte hex>",  // Z^lp_k(x)
      "product_next_eval":       "<32-byte hex>",  // Z^lp_k(x·ω)
      "permuted_input_eval":     "<32-byte hex>",  // A'_k(x)
      "permuted_input_inv_eval": "<32-byte hex>",  // A'_k(x·ω⁻¹)
      "permuted_table_eval":     "<32-byte hex>"   // S'_k(x)
    },
    ...
  ],
  "trash_evals":                     ["<32-byte hex>", ...],  // ntrash entries
  "gwc": {
    "f_commitment": "<48-byte hex>",      // [f]₁ — GWC auxiliary polynomial
    "q_evals":      ["<32-byte hex>", ...],  // nq entries: prover's q_i(x₃)
    "w_commitment": "<48-byte hex>"       // [w]₁ — KZG opening witness π
  }
}
```

**`permutation_product_evals`**: for the last chunk (which opens only at `x` and `x·ω`),
`last_eval` is `null`. For all other chunks, `last_eval` is the evaluation at `x·ω^last`.
The `parsePlutusProof` function flattens this into `prfPermProdEvals` (see §3.5.1).

**`instance_poly_eval`**: omitted from the JSON. midnight-zk does not use committed instances
(col 0 is the zero polynomial, so the eval is always 0). The 32 bytes are still present in
the raw proof byte stream and consumed by the Rust serialiser to keep subsequent offsets
correct, but they are not written to JSON. The Haskell verifier hardcodes 0 at all use
sites — for `instEvals[0]` and for the transcript absorption — so no information is lost.

**`lookup_evals` flatten order** in `prfLookupEvals` (5 per lookup k, in k order):
```
[Z^lp_0(x), Z^lp_0(x·ω), A'_0(x), A'_0(x·ω⁻¹), S'_0(x),
 Z^lp_1(x), Z^lp_1(x·ω), A'_1(x), A'_1(x·ω⁻¹), S'_1(x), ...]
```

### 3.7 Instance JSON

File: `*_plutus_instance.json`. An array of 32-byte LE hex strings, one per public input:

```json
["<32-byte LE hex>", ...]
```

These are the circuit's public inputs (from the public instance column, col 1 in midnight-zk).

## 4. Transcript (Fiat-Shamir)

The transcript is a `BuiltinByteString` state, initialised to `vkTranscriptRepr` (32 bytes).
`absorb state bytes = state <> bytes` (append, no hashing).
`squeeze state = (Scalar (LE(blake2b_256 state) mod q), blake2b_256 state)` — the state is
replaced by the 32-byte hash, not appended.

This matches the midnight-zk Rust prover: `transcript_data = blake2b_256(transcript_data)`.

Absorption order (must exactly match the prover):

```
state₀ = vkTranscriptRepr                            // 32 bytes — circuit identity
state₁ = state₀ <> G1_compressed_zero               // 48 bytes — instance placeholder
state₂ = absorb state₁ (LE_32(len(pubInputs)))
state₃ = foldl (\s n -> s <> LE_32(n)) state₂ pubInputs

state₄ = foldl (<>) state₃ prfAdviceComs            // na × 48 bytes
(θ, state₄s) = squeeze state₄                        // lookup compression challenge

state₅ = foldl (<>) state₄s (interleave prfLookupInputComs prfLookupTableComs)
(β, state₅s) = squeeze state₅                        // permutation/lookup challenge
(γ, state₅ss) = squeeze state₅s

state₆ = foldl (<>) state₅ss prfPermProdComs
state₇ = foldl (<>) state₆ prfLookupProdComs
(trashChal, state₇s) = squeeze state₇               // trash challenge — BEFORE trash coms
state₇t = foldl (<>) state₇s prfTrashComs

state₈ = state₇t <> prfRandomCom
(y, state₈s) = squeeze state₈                        // gate folding challenge

state₉ = foldl (<>) state₈s prfHComs
(x, state₉s) = squeeze state₉                        // shared evaluation point

-- Absorb all evaluations in canonical order:
allEvals = [0] ++ prfAdviceEvals ++ prfFixedEvals ++ [prfRandomEval]
        -- ^ 0: instance_poly_eval hardcoded (always zero in midnight-zk)
        ++ prfPermSigmaEvals ++ prfPermProdEvals ++ prfLookupEvals ++ prfTrashEvals
state₁₀ = foldl (\s n -> s <> LE_32(n)) state₉s allEvals

(x₁, state₁₀s) = squeeze state₁₀                   // within-set combiner
(x₂, state₁₀ss) = squeeze state₁₀s                 // across-set combiner

state₁₁ = state₁₀ss <> prfFCom
(x₃, state₁₁s) = squeeze state₁₁                   // opening point for f

state₁₂ = foldl (\s n -> s <> LE_32(n)) state₁₁s prfQEvalsOnX3
(x₄, _) = squeeze state₁₂                           // fold combiner
```

`interleave [a₀,a₁,...] [b₀,b₁,...] = [a₀,b₀,a₁,b₁,...]`.
`LE_32(n) = integerToByteString LittleEndian 32 n`.

**Critical ordering note:** `trashChal` is squeezed AFTER perm/lookup product coms but
BEFORE the trash coms are absorbed. For circuits with `ntrash = 0`, `prfTrashComs` is
empty, but the squeeze still happens.

## 5. Evaluation points

From the shared challenge `x` (evaluation point at rotation 0):

```
ω_inv  = recip ω                                   // ω⁻¹ = ω^{N-1}
x_next = x * ω                                      // rotation +1
x_prev = x * ω_inv                                  // rotation −1
x_last = x * ω^{-(blinding+1)}                      // last usable row
```

`ω^{-(blinding+1)}` is computed as `(ω_inv)^{blinding+1}` by iterated multiplication
(blinding is small — 5 or 6 in the test circuits).

Domain size: `N = 2^k`. `x^N` is computed by k squarings via `scale N x` which uses
`expModInteger (unScalar x) N q`.

## 6. Gate constraint check

### Why it is needed

The KZG multi-point opening proof checks only that polynomial commitments are consistent with
their claimed evaluations. It does NOT verify that the vanishing quotient commitments
`prfHComs` encode the constraint polynomial `h(X) = Σ constraintᵢ(X) / (X^N − 1)`.

A malicious prover can commit to the zero polynomial for all h-pieces, claim `h(x) = 0`, and
still produce a valid KZG opening — because zero evaluates to zero everywhere. The gate
constraint check closes this gap by verifying:

```
hEval × (x^N − 1)  =  Σᵢ yⁱ × constraintᵢ(x)
```

where `hEval` is derived from the prover's claimed evaluations (via the H-slot assembly in
`assembleRotationSets`) and the right-hand side is the Horner-folded constraint sum.

### 6.1 Expression tree evaluator (`evalGate`)

`evalGate expr advEvals fixEvals instEvals` evaluates a `GateExpr` tree by structural
recursion.  The tree is built once at parse time by `instrsToGateExpr` in `JsonParser.hs`
(see §3.3); no stack or byte-scanning occurs at on-chain evaluation time.

```
evalGate (GEConst s)   adv fix inst = s
evalGate (GEAdv qi)    adv fix inst = adv[qi]
evalGate (GEFix qi)    adv fix inst = fix[qi]
evalGate (GEInst qi)   adv fix inst = inst[qi]
evalGate (GENeg e)     adv fix inst = 0 − evalGate e
evalGate (GEAdd a b)   adv fix inst = evalGate a + evalGate b
evalGate (GEMul a b)   adv fix inst = evalGate a × evalGate b
evalGate (GEScale e s) adv fix inst = evalGate e × s
```

All operand lists (`advEvals`, `fixEvals`, `instEvals`) contain values already lifted to
`Scalar`:

- `advEvals = map mkScalar prfAdviceEvals` — `naq` elements, in `cs.advice_queries()` order
- `fixEvals = map mkScalar prfFixedEvals` — `nfq` elements, in `cs.fixed_queries()` order
- `instEvals = [zero, pubInstEval]` — always exactly 2 elements:
  - index 0: `zero` (hardcoded; col 0 is the zero polynomial in midnight-zk)
  - index 1: public instance column eval, computed via Lagrange interpolation (see below)

**Public instance eval:** midnight-zk places public inputs in instance column 1 (rows 0..np−1).
The verifier interpolates:

```
nInv = recip (mkScalar N)
lagrangeAtRow i = ω^i × (x^N − 1) × nInv × recip(x − ω^i)
pubInstEval = Σᵢ mkScalar(pubInputs[i]) × lagrangeAtRow(i)
```

This computation is exact: the Lagrange basis polynomial `L_i(x)` equals 1 at row i and 0
at all other rows of the domain.

### 6.2 Gate expressions

```
evalE e   = evalGate e advEvals fixEvals instEvals
gateExprs = map evalE (vkGatePolys vk)
```

Each element of `vkGatePolys` is a `GateExpr` tree for one gate constraint polynomial. The
evaluation must equal zero on all active rows for a valid witness; the gate constraint check
verifies this implicitly via the Horner fold and the h-polynomial reconstruction.

### 6.3 Permutation argument constraints

Lagrange polynomials used in the permutation constraints:

```
l0    = lagrangeAtRow 0                             // L_0(x)
lLast = lagrangeAtRow (N − blinding − 1)            // L_{last}(x)
lBlind = Σⱼ₌₀^{blinding−1} lagrangeAtRow (N − blinding + j)
activeRows = 1 − lLast − lBlind
```

For each permutation chunk j (0..np−1), where `np = ceil(npc / chunk_size)`:

```
-- Evaluations (from prfPermProdEvals via the flat layout in §3.5.1):
zj_x    = mkScalar prfPermProdEvals[3j]
zj_xw   = mkScalar prfPermProdEvals[3j+1]
zj_xlast = mkScalar prfPermProdEvals[3j+2]  // only for non-last chunk

-- Coset shifts and column evaluations for columns in chunk j
-- (columns j*chunk_size .. min((j+1)*chunk_size, npc) − 1):
For each column k in chunk j (local_k = 0, 1, ...):
  global_k = j * chunk_size + local_k
  (col_type, eval_idx) = vkPermColTypes[global_k]
  col_eval = advEvals[eval_idx] | fixEvals[eval_idx] | instEvals[eval_idx]
  sigma_eval = mkScalar prfPermSigmaEvals[global_k]
  coset_shift = δ^global_k        // bls12_381_scalar_delta ^ global_k (see §2)
```

Four constraint expressions per chunk:

```
-- (1) Initialisation: z_j(ω^0 · x) = 1 at row 0
e_init = l0 × (1 − zj_x)

-- (2) Finalisation: z_j(x) ∈ {0,1} at last row
e_final = lLast × (zj_x × zj_x − zj_x)

-- (3) Product argument (on active rows):
lhs = zj_xw × Πₖ (col_eval_k + β × sigma_eval_k + γ)
rhs = zj_x  × Πₖ (col_eval_k + β × coset_shift_k × x + γ)
e_prod = activeRows × (lhs − rhs)

-- (4) Linking between chunks (only for j < np−1):
z_next_xlast = mkScalar prfPermProdEvals[3(j+1)+2]  // next chunk at x·ω^last
e_link = l0 × (z_{j+1}(x) − zj_xlast)
```

For the last chunk (j = np−1), only `e_init`, `e_final`, `e_prod` are emitted (no `e_link`).
The Horner fold processes them: for j=0..np−2: `[e_init, e_final, e_prod, e_link]`; for
j=np−1: `[e_init, e_final, e_prod]`.

### 6.4 Lookup argument constraints

For each lookup k (0..nl−1):

```
-- Evaluations:
zpk_x  = mkScalar prfLookupEvals[5k]       // Z^lp_k(x)
zpk_xw = mkScalar prfLookupEvals[5k+1]     // Z^lp_k(x·ω)
Ak_x   = mkScalar prfLookupEvals[5k+2]     // A'_k(x) — permuted input
Ak_xi  = mkScalar prfLookupEvals[5k+3]     // A'_k(x·ω⁻¹)
Sk_x   = mkScalar prfLookupEvals[5k+4]     // S'_k(x) — permuted table

-- θ-compression of input expressions (vkLookupInputExprs[k]):
compInput = foldl (\acc e -> acc × θ + evalE e) 0 (vkLookupInputExprs vk !! k)

-- θ-compression of table expressions (vkLookupTableExprs[k]):
compTable = foldl (\acc e -> acc × θ + evalE e) 0 (vkLookupTableExprs vk !! k)
```

Five constraint expressions:

```
e0 = l0 × (1 − zpk_x)
e1 = lLast × (zpk_x × zpk_x − zpk_x)
e2 = activeRows × (zpk_xw × (Ak_x + β) × (Sk_x + γ)
                 − zpk_x  × (compInput + β) × (compTable + γ))
e3 = l0 × (Ak_x − Sk_x)
e4 = activeRows × (Ak_x − Sk_x) × (Ak_x − Ak_xi)
```

### 6.5 Trash argument constraints

For each trashcan t (0..ntrash−1), where `ntrash = len(vkTrashSelectors vk)`:

```
trash_eval = mkScalar prfTrashEvals[t]
sel = evalE (vkTrashSelectors vk !! t)

-- trashChal-compression of constraint expressions:
compressed = foldl (\acc e -> acc × trashChal + evalE e) 0 (vkTrashConstraintExprs vk !! t)

e_t = compressed − (1 − sel) × trash_eval
```

### 6.6 Horner folding and equality check

All constraint expressions are concatenated in order:

```
allExprs = gateExprs ++ permExprs ++ lookupExprs ++ trashExprs
```

where:
- `permExprs`: chunk 0 exprs, chunk 1 exprs, ..., chunk np−1 exprs (each chunk has 3 or 4 exprs)
- `lookupExprs`: [e0,e1,e2,e3,e4] for lookup 0, then lookup 1, etc.
- `trashExprs`: one expr per trashcan

Horner fold with challenge y:

```
hEvalSum = foldl (\acc e -> acc × y + e) 0 allExprs
```

Equality check (the only circuit-level soundness assertion):

```
hEval × xnMinusOne  ==  hEvalSum
```

where `hEval` is the value assembled for the H slot in `assembleRotationSets` (see §7.2)
and `xnMinusOne = x^N − 1`.

## 7. Rotation set assembly

`assembleRotationSets vk prf specs x x₁ x_next x_prev x_last hEval`

For each `RotationSetSpec` in `specs`:

1. Convert each raw integer rotation offset to a `Rotation` ADT value via `toRotation`
   (conversion happens here because `ccBlinding` is required to identify `RotLast`):

   | Integer offset | `Rotation` | Evaluation point |
   |----------------|-----------|-----------------|
   | `0` | `RotCur` | `x` |
   | `+1` | `RotNext` | `x_next` |
   | `-1` | `RotPrev` | `x_prev` |
   | `-(blinding+1)` | `RotLast` | `x_last` |
   | any other | `RotArb r` | `x · ω^r` (iterated multiply) |

   `RotArb` does not appear in midnight-zk circuits; it is kept for completeness.

2. For each slot (in x₁-power order), with accumulated power `x₁^j`:
   - Look up the commitment and evaluation(s) using the poly-kind dispatch below.
   - Add `x₁^j × commitment` to the set's combined commitment (`rsScaledComs`).
   - For each rotation point `pts[r]`, add `x₁^j × eval_at_rotation_r` to `rsQEvalsAtPts[r]`.

3. Return `RotationSet { rsPoints, rsScaledComs, rsQEvalsAtPts }` where `rsPoints` are the
   resolved evaluation points.

The H slot (kind `"H"`) is the only slot that expands to multiple physical commitments (see §7.2).

### 7.1 Poly-kind dispatch table

| Kind | Commitment | Eval at rotation `r` |
|------|-----------|----------------------|
| `Advice` | `uncompress prfAdviceComs[ssIndex]` | `mkScalar prfAdviceEvals[ssEvalIdxs[r]]` |
| `Instance` | `G1_zero` (uncompressed identity) | `zero` (hardcoded) |
| `LookupTable` | `uncompress prfLookupTableComs[ssIndex]` | `mkScalar prfLookupEvals[5k+4]` |
| `Trash` | `uncompress prfTrashComs[ssIndex]` | `mkScalar prfTrashEvals[ssIndex]` |
| `Fixed` | `uncompress vkFixedComs[ssIndex]` | `mkScalar prfFixedEvals[ssEvalIdxs[r]]` |
| `PermSigma` | `uncompress vkPermSigmaComs[ssIndex]` | `mkScalar prfPermSigmaEvals[ssIndex]` |
| `H` | (see §7.2) | `hEval` |
| `Random` | `uncompress prfRandomCom` | `mkScalar prfRandomEval` |
| `PermProd` | `uncompress prfPermProdComs[ssIndex]` | `RotCur` → `prfPermProdEvals[3j]`; `RotNext` → `[3j+1]`; `RotLast` → `[3j+2]` |
| `LookupProd` | `uncompress prfLookupProdComs[ssIndex]` | `RotCur` → `prfLookupEvals[5k]`; `RotNext` → `[5k+1]` |
| `LookupInput` | `uncompress prfLookupInputComs[ssIndex]` | `RotCur` → `prfLookupEvals[5k+2]`; `RotPrev` → `[5k+3]` |

`uncompress` = `bls12_381_G1_uncompress`.

### 7.2 H-piece special case

> **Why H appears in the rotation set at all:**
> `computeHEval` (§6.2) computes what `h(x)` *must* equal — the Horner-folded gate/perm/lookup
> constraint sum divided by `x^n − 1`. That is the verifier's own calculation of the correct
> value. But `prfHComs` are the prover's opaque commitments to the actual `h` polynomials they
> used; the verifier cannot tell from the G1 points alone whether they encode the right `h`.
> Including H in the rotation set with `hEval` as its evaluation side is what ties the two
> together: the final KZG pairing check enforces that the committed polynomials open to exactly
> `hEval`. A prover who committed to a fraudulent `h` would produce a commitment that opens to
> a different value, failing the pairing check.

The vanishing quotient `h(X) = h₀(X) + X^{N−1} · h₁(X) + X^{2(N−1)} · h₂(X) + ...` occupies ONE
logical x₁ slot (slot index `hSlotPos`) in the rotation set, but expands to `nh` physical
G1 commitments `prfHComs[0..nh−1]`.

The evaluation point `hSplit = x^{N−1}` is used to reconstruct the logical slot evaluation:

```
h(x) = Σⱼ₌₀^{nh−1} hSplit^j × hⱼ(x)   ← not needed: hEval is already assembled
```

The combined commitment for the H slot, with x₁ factor `x₁^hSlotPos`:

```
x₁^hSlotPos × Σⱼ₌₀^{nh−1} hSplit^j × [hⱼ]₁
```

is computed as an MSM: `msm (zipWith (×) x₁_powers hSplit_powers) prfHComs` where
`x₁_powers[j] = x₁^hSlotPos × hSplit^j`.

The evaluation contribution is simply `x₁^hSlotPos × hEval` (one scalar, added to
`rsQEvalsAtPts[0]` — the H slot always appears in the single-rotation set `{x}`).

## 8. Generic GWC multi-point opening

`verifyGwc sG2 fCom x₂ x₃ x₄ rotSets qEvalsOnX3 piPt`

Given the assembled `[RotationSet]` and the challenges from the transcript:

**Step 1: Lagrange interpolants**

For each rotation set i, compute `r_i(x₃)` — the unique polynomial of degree < |Sᵢ| that
passes through the points `{(rsPoints[k], rsQEvalsAtPts[k])}`:

```
r_i(x₃) = lagrange rsPoints rsQEvalsAtPts x₃
```

Lagrange interpolation for `m` points:

```
lagrange pts evals z = Σₖ evals[k] × Πⱼ≠ₖ (z − pts[j]) / (pts[k] − pts[j])
```

**Step 2: Vanishing polynomial evaluations**

```
V_i(x₃) = Πₖ (x₃ − rsPoints[k])
```

**Step 3: GWC contributions**

```
c_i = (qEvalsOnX3[i] − r_i(x₃)) × recip(V_i(x₃))
```

**Step 4: f(x₃)**

```
f(x₃) = Σᵢ x₂^i × c_i      (Horner: horner x₂ cs)
```

**Step 5: Combined commitment and evaluation**

```
finalCom = Σᵢ x₄^i × qCom_i  +  x₄^m × fCom
vEval    = Σᵢ x₄^i × qEvalsOnX3[i]  +  x₄^m × f(x₃)
```

where `qCom_i = Σⱼ rsScaledComs_i[j]` (the summed pre-scaled commitments for set i).

In Plutus these MSMs use `bls12_381_G1_multiScalarMul`.

**Step 6: KZG pairing** (see §9)

## 9. KZG pairing check

The KZG opening relation `f(x₃) = eval` for commitment `[f]₁ = finalCom` with witness
`π = piPt`:

```
e(piPt, sG2) = e(finalCom − vEval·G1_gen + x₃·piPt, G2_gen)
```

In Plutus:

```
ml1 = bls12_381_millerLoop piPt sG2
ml2 = bls12_381_millerLoop (finalCom - vEval·G1_gen + x₃·piPt) G2_gen
bls12_381_finalVerify ml1 ml2
```

The right-hand G2 argument is the **positive** generator `bls12_381_G2_compressed_generator`
(decompressed). Using its negation would give False.

## 10. Circuit-specific parameters

All 10 test circuits use `cs_degree = 5` (so `chunk_size = 3`, `nh = 4`).
`blinding_factors` is 5 for four circuits and 6 for the other six — see the table below.

blinding commitment points in the proof. It is derived from the actual proof byte length
rather than from a fixed formula, so circuits with the same `blinding_factors` can have
different `ntrash` values (e.g. Ethereum ECDSA has `blinding_factors = 6` but `ntrash = 0`).

| Bitcoin Schnorr | 5 | 9 | 7 | 12 | 4 | 25 | 42 | 0 | 11 | 5 |
| Poseidon hash | 6 | 8 | 1 | 8 | 3 | 11 | 19 | 1 | 8 | 4 |
